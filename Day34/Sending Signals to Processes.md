# 1. What is a Process?

A **process** is simply a **program that is currently running**.

For example:

```bash
firefox
```

starts a Firefox process.

```bash
sleep 500
```

starts a sleep process.

Every running process has its own:

* Process ID (PID)
* Parent Process ID (PPID)
* Memory allocation
* CPU usage
* State
* Open files
* Environment variables

Linux manages every running program as a process.

---

# 2. What is a Signal?

A **signal** is a software interrupt sent to a running process.

Think of it as a message telling the process to perform an action.

Examples include:

* Stop running
* Continue running
* Reload configuration
* Terminate
* Produce a core dump

Unlike hardware interrupts, signals are generated by software.

---

## Real-Life Analogy

Imagine a worker in an office.

| Office Situation                    | Linux Equivalent |
| ----------------------------------- | ---------------- |
| Manager asks worker to stop         | SIGTERM          |
| Security removes worker immediately | SIGKILL          |
| Worker takes a break                | SIGSTOP          |
| Manager asks worker to continue     | SIGCONT          |
| Fire alarm                          | Signal           |

A signal is simply an instruction sent to the process.

---

# 3. How Signals Work

```
Application Running
        │
        │
Signal Sent
        │
        ▼
Kernel Receives Signal
        │
        ▼
Kernel Delivers Signal
        │
        ▼
Program Receives Signal
        │
        ├── Ignore
        ├── Handle
        └── Default Action
```

The Linux kernel delivers the signal to the target process.

The process then decides whether to:

* Ignore it
* Handle it with custom code
* Perform the default action

---

# 4. Default Signal Actions

Every signal has a default behavior.

There are three common actions.

---

## A. Terminate

The process exits.

Example

```
sleep 1000
```

Press

```
Ctrl+C
```

Output

```
Terminated
```

---

## B. Stop

The process pauses.

Example

```
Ctrl+Z
```

Output

```
Stopped
```

The process still exists in memory.

---

## C. Core Dump

Linux saves the memory image of the application for debugging.

Used mainly by developers.

# 5. Important Linux Signals

---

## 5.1 SIGTERM (15)

### Meaning

Terminate gracefully.

### Purpose

This is the preferred way to stop a process.

Linux asks:

> "Please finish your work and exit."

The process gets time to:

* Save data
* Close files
* Disconnect clients
* Release memory
* Write logs

### Example

```
kill 4567
```

or

```
kill -TERM 4567
```

---

### Real Example

Stopping Apache

```
kill -TERM 2145
```

Apache:

* finishes active requests
* closes sockets
* saves logs
* exits safely

---

# 5.2 SIGKILL (9)

### Meaning

Force kill.

Unlike SIGTERM,

the process cannot:

* ignore it
* catch it
* block it

Linux immediately removes the process.

Example

```
kill -9 4567
```

---

### Why avoid SIGKILL?

Suppose MySQL is writing data.

```
kill -9 mysql
```

The database is stopped immediately.

Possible consequences:

* incomplete writes
* corrupted transactions
* lost temporary data

Therefore:

**Always try SIGTERM first.**

# 5.3 SIGINT

Generated by

```
Ctrl+C
```

Interrupts the foreground program.

Example

```
sleep 1000
```

Press

```
Ctrl+C
```

Output

```
^C
```

The program exits.

---

# 5.4 SIGQUIT

Generated by

```
Ctrl+\
```

Similar to Ctrl+C

but additionally creates a

**Core Dump**

Used for debugging.

---

# 5.5 SIGSTOP

Stops a process immediately.

Cannot be ignored.

Example

```
kill -STOP PID
```

The process enters

```
T
```

state.

Resume using

```
kill -CONT PID
```

---

# 5.6 SIGTSTP

Generated by

```
Ctrl+Z
```

Difference

SIGSTOP

cannot be ignored.

SIGTSTP

can be handled.

Example

```
vim file.txt
```

Press

```
Ctrl+Z
```

Output

```
Stopped
```

Resume

```
fg
```

---

# 5.7 SIGCONT

Continues a stopped process.

Example

```
kill -CONT PID
```

or

```
fg
```

---

# 5.8 SIGHUP

Originally meant

Hang Up.

Old UNIX systems used terminals connected by telephone lines.

When the connection closed,

the terminal generated

```
SIGHUP
```

Today

many services interpret SIGHUP as

"Reload your configuration."

Example

```
kill -HUP apache_PID
```

Apache reloads its configuration without restarting.

# 10. Using pkill

Instead of PID,

use process name.

Example

```
pkill firefox
```

Kills all firefox processes.

Specific signal

```
pkill -HUP nginx
```

Reload nginx.

---

Kill by user

```
pkill -u bob
```

---

Force

```
pkill -9 -u bob
```

---

Kill by terminal

```
pkill -t tty2
```

Terminates every process on tty2.

---

Kill children of parent

```
pkill -P 500
```

Kills only child processes.

Parent survives.

---

# 11. Using killall

Terminate all processes having the same name.

```
killall firefox
```

Force

```
killall -9 firefox
```

Difference

```
kill
```

needs PID.

```
killall
```

needs process name.

# 15. Signal Priority (Best Practice)

Always follow this order:

```
SIGTERM
     ↓
SIGINT
     ↓
SIGKILL
```

### Step 1 – SIGTERM

Allows graceful shutdown.

### Step 2 – SIGINT

Interrupts if still running.

### Step 3 – SIGKILL

Forcefully kills the process if it refuses to terminate.

This is the sequence recommended by Red Hat because it minimizes the risk of data loss or corruption.

# 16. Command Comparison

| Command          | Purpose                               | Example           |
| ---------------- | ------------------------------------- | ----------------- |
| `kill PID`       | Send signal to a specific PID         | `kill 1234`       |
| `kill -9 PID`    | Force kill a process                  | `kill -9 1234`    |
| `kill -HUP PID`  | Reload configuration                  | `kill -HUP 1234`  |
| `kill -STOP PID` | Pause a process                       | `kill -STOP 1234` |
| `kill -CONT PID` | Resume a paused process               | `kill -CONT 1234` |
| `pgrep name`     | Find PID by process name              | `pgrep sshd`      |
| `pgrep -l name`  | Show PID and process name             | `pgrep -l sshd`   |
| `pkill name`     | Kill processes by name                | `pkill firefox`   |
| `pkill -u user`  | Kill all processes of a user          | `pkill -u bob`    |
| `pkill -t tty3`  | Kill all processes on a terminal      | `pkill -t tty3`   |
| `pkill -P PID`   | Kill child processes of a parent      | `pkill -P 2100`   |
| `killall name`   | Kill all processes with the same name | `killall firefox` |
| `jobs`           | List background jobs                  | `jobs`            |
| `who`            | Show logged-in users                  | `who`             |
| `w`              | Show users and their current activity | `w`               |
| `pstree -p`      | Display process hierarchy             | `pstree -p`       |

# RHCSA EX200 Exam Tips

* **SIGTERM (15)** is the default signal sent by `kill`; it allows the application to shut down cleanly.
* **SIGKILL (9)** cannot be blocked, ignored, or handled—use it only as a last resort.
* **Ctrl+C** sends **SIGINT**, while **Ctrl+Z** sends **SIGTSTP** to suspend the foreground process.
* Use **`pgrep`** to locate PIDs and **`pkill`** when you want to signal processes by **name**, **user**, **TTY**, or **parent PID**.
* Use **`killall`** to signal every process with the same command name.
* `jobs`, `fg`, and `bg` manage jobs started from the current shell.
* `pstree` helps visualize parent–child relationships before terminating related processes.
* Remember the safest shutdown order: **SIGTERM → SIGINT → SIGKILL**.

### Quick Memory Trick

* **TERM** → *Ask the process politely to exit.*
* **INT** → *Interrupt the process.*
* **KILL** → *Force the process to stop immediately.*
